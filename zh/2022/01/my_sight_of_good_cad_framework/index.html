3 Good Practices of Hardward CAD Framework

  2022-01-09   THOUGHT   PM Learning  评论 字数统计: 2308字 阅读时长: 5分  English Version

在IC（集成电路）行业，CAD是一种使硬件设计者的生活变得简单的功能。这就是我对硬件设计中的CAD的定义。“容易 “包含不同的含义。

  • 易于理解输入
  • 易于使用功能
  • 易于检查结果(准确和可靠)

很多人总是喜欢用苹果产品作为例子。我在这里。

苹果有合适的技术，但不是最先进的技术。

最先进的技术意味着更多的风险。但是，把最适合客户的优越技术放在一起，是成功的关键。

但是，无论是否遵循以下原则，最好的解决方案永远是最适合场景的方案。

CAD框架是为专业人士设计的

所有的硬件设计师都是行业内的专家。当我第一次加入CAD小组时，我在想如何帮助，我可以帮助专业人士什么。

我需要找出我对CAD工具用户的价值，对我们的专业硬件设计师的价值，因为在制作芯片方面我并不比他们强。

由于VLSI CAD实用程序的客户是专业的硬件工程师，CAD实用程序的目标是开发一套能够帮助提高准确性、生产力的工具。

  • 容易获得数据
  • 易于发送数据
  • 易于使用数据

在一个大组织中，容易也意味着标准化。

良好做法1：最小的惊奇

a.k.a. 最少的惊讶

配置总是一个很好的例子来解释这个规则。

在CAD工具中，我们可能有多种方式来配置操作。

  • 其中一些是来自环境变量
  • 有些是来自于配置文件，如（yaml，json）。
  • 有些是来自项目钩子、默认值等。
  • 有些是来自于移交标准
  • 有些是来自EDA工具本身。

有时我们无法避免引入混乱。那么问题就变成了如何解决这个问题，清除混乱。

单一真实来源(SSoT)

而在我们的配置文件中，我们可能有一个if-else语句来根据条件进行切换。

如果学习曲线如此陡峭，就不适合新项目。他们会拒绝迁移。

即使工程师掌握了一次实用程序，他们也会在下一个项目中再次学习。

理想的情况是，在我们的平台上，所有的实用程序都可以从一个来源读取配置。

但这是不实际的。硬件设计还没有达到人工智能的要求。在日常工程工作中，我们仍然需要处理大量的案例。

所以解决方案是，我们可以将功能拆分，或者将功能逐层模块化。每一层，我们都有API，或者我们称之为配置文件，用于输入和输出。

输入和输出在一个条目文件中被很好地定义。这个文件被称为 “Metadata”，API应该始终如一地提供 “Metadata”，而不是猜测。

{
  "metadata" : {
    "name" : "flow_config",
    "user": "icsteve",
    "$rel": {
       "connect" : "connect_api_1"
    }
  },
  "数据集" : {
    "width" : 100,
    "高度" : 100
  }
}

在真实场景中，我们可以有更多的约束。比如，数据集必须是对象，k-v对。

我们不把这个规则理解为一个单一的文件，而是带来了元数据的概念，它代表了 “单一来源”。

而在元数据中，我们还需要描述每个元数据键之间的关系。这也可以是元数据的一部分。

关系（$rel）是某些业务需求的逻辑，所以需要产品SPEC来描述如何使用和解释关系。

无歧义

在编写需求/模式或配置时，“毫不含糊 “总是被谈及。

有些配置使用on|off或1|0就很好，那么我们应该保持简单。

有时1或0是不知道要做什么的，所以我们可能需要精确地使用这个词。

在定义模式或设计配置时，最好是把它发给不同的人审阅。 如果有任何问题进入你的邮箱，那说明这个设置不够清楚，会发生什么。

###约定与配置

我更喜欢明确的配置，而不是惯例。

但惯例（默认命名）可以使配置变得相当简单。

良好做法2：引导而不是指责

当用户提出使用问题，或者他们滥用一些选项时。

你是否只是回答。“请阅读文档！！！”

不！不！! 不!!!

大多数用户都是使用EDA（电子设计自动化）工具的资深用户。

如果用户不清楚，那就意味着__目前的实用设计理念很糟糕！__。

我总是建议新来的CAD人员参加几个项目的tapeout。然后我们就可以得到我们的客户所面临的真实场景。

当然，软件和硬件有相同的基础。但是，仍然有一些DDD（领域驱动设计）的差异。

实例和模板

提供足够的模板和例子，使用户尽可能快地进入。

模板是必不可少的。用户能够从模板（菜谱）开始，这些模板是经过CAD开发人员测试的。

为了教育用户，我们应该怎么做？

HowTo

开发一个培训材料，帮助用户建立知识。

WhatNext

当一个错误发生时，尝试给出。

  • 发生了什么，错误信息，错误代码，等等。
  • 错误代码的位置，如果可能的话突出显示
  • 给出建议，下一步是什么

Dryrun

我们有数以千计的配置项目，要记住所有的项目并不容易。

因此，审查适用于即将到来的运行的设置是非常重要的，帮助用户检查和识别潜在的问题。

这就是所谓的 “干运行 “的能力。

干运行日志是另一种可能非常有用的工作艺术。

帮助系统

帮助系统不仅仅是指文件和培训。

文件可以是代码（在日志和运行时显示）。

你可以有一个100页的word/pdf文档，但要考虑用户如何使用这个文档。

帮助系统还包括-h/-help、man page、错误处理、变量追踪等。

帮助系统是一个很大的话题。我们将在下周谈论它帮助系统。

良好实践3：保持简单，愚蠢

解耦，多即是少

我们有一些情况需要重复使用一些工具或代码，那么模块化就是一个很好的方法。

想一想这种情况，这个库到处都在使用，有一天有一个bug被引入，导致整个系统瘫痪。

当然，我们还有其他方法可以回滚。而在我们的方案中，可重用性确实是需要的。

如果我控制修订版，并且对每个模块单独进行检查，会怎么样？那么，即使我想为一个特定的应用程序更新通用工具，也不会影响其他的。

功能选项

参数或选项不应该难以理解，也不应该太多。

如果需要很多参数，该工具可能也需要考虑使用输入配置文件。

视觉化

CLI是首先需要的，但是GUI或可视化也是很好的选择。

这意味着网络技术是CAD职业生涯中需要获得的一项好技能。
