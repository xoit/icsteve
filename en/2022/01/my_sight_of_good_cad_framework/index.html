3 Good Practices of Hardward CAD Framework

Jan 9, 2022 6 min read Learning PM

In IC (Integrated Circuit) industry, CAD is a function to make hardware designer’s life easy. This is how I define CAD in hardware design. “Easy” contains different meanings.

  • Easy to understand the inputs
  • Easy to use the functions
  • Easy to check the result (Accurate and reliable)

A lot of people always like to use Apple products as an example. Here I am.

Apple has suitable technologies but not the most advanced technology.

The most advanced technology means more risk. But to put the superior technology which is best fit the customer is the key to success.

BUT, the best solution is always the one that fits the scenario best no matter it follows the following principles or not.

CAD Framework is designed for the professional

All the hardware designers are experts in the industry. When I first joined the CAD group, I was thinking about how to help and what can I help the professionals.

I need to find out my value to the CAD utility users, to our professional hardware designers since I am not better than them in terms of making chips.

Since the customers of VLSI CAD utilities are professional hardware engineers, the goal of CAD utilities is to develop a set of tools that can help with accuracy, productivity.

  • Easy to get the data
  • Easy to send the data
  • Easy to use the data

In a big organization, easy also means standardization.

Good practice 1: Least Astonishment

a.k.a. least surprise

Configuration is always a good example to explain this rule.

In CAD utility, we may have multiple ways to configure the operations.

  • Some of them are from environmental variables
  • Some of them are from the configuration file, like (yaml, json)
  • Some of them are from project hook, defaults, etc.
  • Some of them are from handover criteria
  • Some of them are from the EDA tool itself

Sometimes we can not avoid introducing confusion. Then the problem becomes how to resolve the issue, clear the confusion.

Single Source of True (SSoT)

And in our configuration file, we may have an if-else statement to switch based on the conditions.

If the learning curve is so steep, it is not suitable for new projects. They will refuse to migrate.

Even the engineers master the utility once, they will learn again in the next project.

The ideal case is that, in our platform, all the utilities can read the configuration from one source.

But this is not practical. The hardware design is not AI-ready yet. We still need to handle a lot of cases during daily engineering work.

So the solution is that we can split the function or modularize the function layer by layer. Each layer, we have the APIs, or we call configuration files, for inputs and outputs.

The inputs and outputs are well defined in one entry file. The file is called “Metadata”, and the APIs should consistently deliver with “Metadata”, no guessing.

{
  "metadata" : {
    "name" : "flow_config",
    "user": "icsteve",
    "$rel": {
       "connect" : "connect_api_1"
    }
  },
  "dataset" : {
    "width" : 100,
    "height" : 100
  }
}

In the real scenario, we can have more constraints. Like, the dataset must be the objects, k-v pairs.

Instead of understanding this rule as a single file, we bring the metadata concept, which represents the “Single Source”.

And in metadata, we also need to describe the relationship between each metadata key. This could be part of the metadata as well.

The relationship ($rel) is the logic for certain business needs, so it requires product SPEC to describe how to use and explain the relations.

Unambiguous

The “unambiguous” is always talked about when writing a requirement/schema or configuration.

Some configuration is good to use on|off or 1|0, then we should keep it simple.

Sometimes 1 or 0 is no telling what is going to do, so we might need to use the word precisely.

When defining schema or design configuration, it is better to send it out to different people for review. If any questions come into your email box, that means the setting is not clear enough what will happen.

Conventions vs Configuration

I prefer explicit configuration, not conventions.

But the conventions (default naming) can make the configuration pretty simple.

Good practice 2: Guide instead of Blaming

When users are asking usage questions, or they abuse some options.

Did you just reply: “Please read the document!!!”

No! No!! No!!!

Most users are experienced users using the EDA (Electronic Design Automation) tools.

If the user is not clear, that means The philosophy of current utility design is terrible!

I always suggest the new CAD comers participate in several project tapeout. Then we can get the real scenario our customers are facing.

Of course, Software and hardware share the same ground. But still, there are some DDD (domain-driven design) differences.

Examples and Templates

Provide enough templates and examples to bring the users in as quickly as possible.

The template is essential. Users are able to start with the templates (recipes), which are tested by the CAD developers.

To educate the user, what should we do?

HowTo

Develop a training material to help users to build knowledge.

WhatNext

When an error happened, try to give out:

  • What happened, error message, error code, etc.
  • Error code location, highlight if possible
  • Give out the suggestion, whatnext

Dryrun

We have thousands of configuration items, it is not easy to remember all of them.

So to review the settings which will apply to the upcoming run is essential, helping the users to check and identify the potential issues.

This is called the ability of “dry-run”.

Dry-run log is another art of work that could be very useful.

Help System

The help system is not only about documents and training.

The document could be in code (show during logging and runtime)

You could have a 100-page word/pdf document, but think about how do the users use the document.

The help system also includes -h / --help, man page, error handling, variable tracing, etc.

The help system is a big topic. We will talk about it next week Help System.

Good practice 3: Keep it Simple, Stupid

Decouple, more is less

We have some cases that we need to re-use some utility or code, and then modularization is a good approach.

Think about this case, the library is used everywhere, and one day there is a bug introduced which causes the whole system down.

Of course, we have other ways to roll back. And reusability is really required in our scenario.

What if I control the revision, and for each module, it is checked out separately. Then even I want to update the common utility for a specific application, it will not affect others.

Functional Options

The argument or option should not be hard to understand and should not be too many.

If there are many arguments needed, the utility may need to consider use input configuration files as well.

Visualization

CLI is needed for the first place, but GUI or Visualization is also great to have.

That means web technology is a good skill to gain in the CAD career.
