Good Hardward CAD Framework In my mind

  2022-01-08   THOUGHT   PM Learning  Comments Word Count: 746words Read Count: 4minutes  中文版

In IC (Integrated Circuit) industry, CAD is a function to make hardware designer’s life easy.

This is how I define the CAD in the hardware design. “Easy” contains different meanings.

  • Easy to understand the inputs
  • Easy to use the functions
  • Easy to check the result (Accurate and reliable)

A lot of people always like to use Apple products as an example. Here I am.

Apple has good technologies, but not the most advanced technology.

The most advanced technology means more risk. But to put the good technology which is best fit the customer is the key of success.

BUT, the best solution is always the one which fits the scenario best no matter it follows the following principles or not.

CAD Framework is designed for the professional

All the hardware designers, they are experts in the industry. When I first joined the CAD group, I was thinking how to help and what can I help for the professionals.

I need to find out my value to the CAD utility users, to our professional hardware designers since I am not better than them in terms of making chips.

Since the customer of VLSI CAD utilities is professional hardware engineers, the goal of CAD utilities is to develop a set of tools which can help on the accuracy, productivity.

  • Easy to get the data
  • Easy to send the data
  • Easy to use the data

In big organization, easy also means standardization.

Good practise 1: Least Astonishment

a.k.a. least surprise

Configuration is always a good example to explain this rule.

In CAD utility, we may have multiple ways to configure the operations.

  • Some of them are from environment variables
  • Some of them are from configuration file, like (yaml, json)
  • Some of them are from project hook, defaults, etc.
  • Some of them are from handover critials
  • Some of them are from EDA tool itself

Sometimes we can not avoid introducing confusion, then the problem becomes how to resolve the issue, clear the confusion.

Single Source of True (SSoT)

And in our own configuration file, we may have if-else statement to switch based on the conditions.

If the learning curve is so steep, it is not good for new projects, they will refuse to migrate.

Even the engineers master the utility once, they will learn again in the next project.

The ideal case is that, in our platform, all the utilities can read the configuration from one source.

But this is not practical. The hardware design is not AI-ready yet, we still need to handle a lot of cases during daily engineering work.

So the solution is that we can split the function or modularize the function layer by layer. Each layer, we have the APIs, or we call configuration files, for inputs and outputs.

The inputs and outputs are well defined in one entry file. The file is called “Metadata”, and the APIs should always deliver with “Metadata”, no guessing.

{
  "metadata" : {
    "name" : "flow_config",
    "user": "icsteve",
    "$rel": {
       "connect" : "connect_api_1"
    }
  },
  "dataset" : {
    "width" : 100,
    "height" : 100
  }
}

In the real scenario, we can have more constraints. Like, the dataset must be the objects, k-v pairs.

Instead of understanding this rule as a single file, we bring the metadata concept which represents the “Single Source”.

And in metadata, we also need to describe the relationship between each metadata key, this could be part of the metadata as well.

The relationship ($rel) is the logic for certain business needs, so it requires product SPEC to describe how to use and explain the relations.

Unambiguous

The “unambiguous” is always talked when writing a requirement/schema or configuration.

Some configuration is good to just use on|off, or 1|0, then we should keep it simple.

But some times, 1 or 0 is not really telling what is going to do, so we might need to use the word specifically.

When we define schema, or design configuration, it is better to send it out to different people for review. If a lot of questions come into your email box, that means the setting is not clear enough what is going to happen.

Conventions vs Configuration

I prefer explicit configuration, not conventions.

But the conventions (default naming) can make the configuration pretty simple.
