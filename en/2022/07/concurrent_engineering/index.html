Concurrent Engineering in ASIC

  2022-07-06   ASIC   Concurrent Engineering ASIC  Comments Word Count: 696words Read Count: 4minutes  中文版

“Concurrent Engineering” is a systematic method of designing and developing products (projects) where different activities run simultaneously.

This decreases the production time leading to reduced costs. Although it has more challenges on extensive communication and coordination.

My Big Point: Apparently, the CAD flow is trying to finish one specific task. Actually, the CAD flow is building a comunication channel between CAD and Designer, between Designer and EDA tool.

“Concurent Engineering” has followingbenefits:

  • Less material wasted and less engineering time
  • Short time to tapeout and time to market
  • Allows for early correction of design, flow, and eda tools
  • Large amount of same methodology produced in a shorter amount of time
  • Ability to leverage teamwork and make infomred decisions
  • Better communication and project management

Concurrent Engineering in ASIC

In the ASIC flow, we are already practising Concurrent Engineering a lot.

None of the companies will follow the sequential process. Time to market is as important as design quality, even more.

Partition is a way to parallel the work and we can put sub-blocks together on the top. (This is not the only reason of partition, EDA capacity is another important reason)

So when we finish 60% (or 80%) of RTL design, we will start to Flow flushing in order to verify the ASIC methodology earlier.

My Big Point: Experience or knowledge passing from one place to another is super important in the ASIC design.

And meanwhile, a lot of work in specific area could be split to small jobs and we can pull more resource into the project and make it concurrent.

Some of the verification or other work, like timing, power, DFT, etc. we will do all the way through the whole project. When we build the CAD flow or plan the resource, we need to pay attention to these types of work.

Mock tapeout could let the foundry get the manufacture verification earlier and reduce the risk when doing the real tapeout.

Partition

The partition is a great way for concurrent engineering idea.

There are several reasons for the partition approach.

  • The complexity of certain design logic, some of them are analog, some of them might be RF, etc.
  • The EDA tool capacity limitation. The EDA tool can not handle millions of millions gates design.
  • Make full use of resource to reduce the time to tapeout.

Flow flushing

This is very common, we are trying to automate our flow as much as possible. So we need to flush the flow before running the 100% good data.

Like tune the tool settings, pick the strategy based on the design type.

As soon as the library, tech files are ready, we can start to flush the flow with small test case.

Even the library is not ready,

Mock Data

Use mock data is a common way to work parallelly.

  • Mock tech files, like use the old one and add some new rules
  • Mock library, or empty .lib, like verilog to lib.
  • Mock design, small test case, design from previous revision
  • Mock tapeout, just list all the masks and chip size, etc.
  • Mock API, work around some critical issues.

More detail in specific process

There are a lot of more details in one specific area. Like logic synthesis and physical implementation, we can do a lot of work to apply the concurrent engineering concept.

I may have separate post for the topics when dive into each flow.

Concurrent Engineering in CAD

CAD flow could help on concurrent engineering mindset as well and also plays an important role.

Optimize the dependency

The dependency in triditional ASIC flow is “stage” based. Like RTL, DFT, SYN, APR-Floorplan, APR-Placement, APR-CTS, APR-PostCTS, APR-Routing, APR-PostRouting, Extraction, STA, PhysicalVerification, EMIR, etc.

But when we look at our daily work, we are spending a lot of time to do the iterations, debugging, reporting, these activities are not needed for the downstream processes.

The downstream jobs only need the output collateral as the input, that means the downstream flow may only need verilog to check the logic equivalent, or only need saved database but not reporting, output data, etc.
